_WrappedCtx Wrapper Analysis
=============================

Background
----------
PR https://github.com/pytorch/pytorch/pull/174079 changes ctx.needs_input_grad
from a stored tuple to a dynamic property that recomputes on each access using
task_should_compute_output(). This allows custom autograd.Function backward
methods to see which gradients are actually needed when inputs= is specified
in backward().

Because the getter is now dynamic, code that previously mutated
ctx.needs_input_grad (in torch/_library/autograd.py) can no longer do so --
the next read would bypass the stored value. A _WrappedCtx wrapper is
introduced that intercepts needs_input_grad reads while forwarding everything
else to the inner ctx.

Implementation
--------------
    class _WrappedCtx:
        def __init__(self, inner_ctx, needs_input_grad):
            self.__dict__["_inner_ctx"] = inner_ctx
            self.__dict__["_needs_input_grad"] = needs_input_grad

        @property
        def needs_input_grad(self):
            return self._needs_input_grad

        def __getattr__(self, name):
            return getattr(self._inner_ctx, name)

        def __setattr__(self, name, value):
            setattr(self._inner_ctx, name, value)

Scope
-----
The wrapper is only used in torch/_library/autograd.py for library-registered
custom ops (via torch.library). It is NOT used for user-written
autograd.Function subclasses -- those get the dynamic getter directly on the
real ctx.

Public Surface Audit
--------------------
The real ctx is a THPFunction (C extension type) with the MRO:
FBackward -> BackwardCFunction -> _FunctionBase -> FunctionCtx -> _HookMixin -> object

Every attribute/method accessible from Python is listed below with whether the
wrapper handles it correctly.

Object Protocol
---------------
  isinstance(ctx, FunctionCtx)    WRONG  returns False          risk: MEDIUM
  type(ctx)                       WRONG  returns _WrappedCtx    risk: LOW
  vars(ctx) / ctx.__dict__        OK     works (no __slots__)
  dir(ctx)                        WARN   shows _WrappedCtx attrs  risk: VERY LOW
  repr(ctx) / str(ctx)            WARN   default object.__repr__  risk: VERY LOW
  id(ctx) / is                    WARN   different identity       risk: VERY LOW
  weakref.ref(ctx)                OK     works (has __weakref__)
  hash(ctx)                       OK     default id-based hash
  bool(ctx)                       OK     returns True
  == / !=                         WARN   identity-based           risk: VERY LOW
  pickle                          WRONG  fails (real ctx also can't be pickled)
  copy.copy()                     WARN   copies wrapper, not inner  risk: VERY LOW
  dunder methods (__getitem__ etc) N/A   not defined on THPFunction

Top Risks
---------
1. isinstance checks (Medium): isinstance(ctx, FunctionCtx) returns False.
   Mitigable by adding a __class__ property that returns type(self._inner_ctx).
   This is the only realistic risk.

2. type() checks (Low): type(ctx) is BackwardCFunction returns False. Not
   fixable without a C extension type. However, type() exact checks are
   uncommon -- isinstance is the standard idiom.

3. Wrapper is only used for torch.library ops (context): The wrapper only
   applies to ops registered via torch.library.register_autograd. User-written
   autograd.Function subclasses receive the real ctx with the dynamic getter
   directly -- no wrapper involved. This significantly limits the blast radius.

Decision Point
--------------
Should we add __class__ property forwarding?

    @property
    def __class__(self):
        return type(self._inner_ctx)

This makes isinstance transparent. The tradeoff is that it makes the wrapper
subtler to debug -- isinstance says it's a BackwardCFunction but type() says
it's _WrappedCtx.
