What is the difference between _WrappedCtx and functorch's WrappedCtx, and should we combine them?
====================================================================================================

There are two ctx wrapper implementations in the codebase:

  1. _WrappedCtx          in torch/_library/autograd.py       (new, this PR)
  2. WrappedCtx           in torch/_functorch/autograd_function.py  (existing)


Functorch's WrappedCtx
----------------------
Base class for wrapping ctx in functorch transforms (vmap, grad).
Uses a _pt_reserved_attrs mechanism to distinguish internal attrs from
forwarded ones.

    class WrappedCtx:
        _pt_reserved_attrs = ("_pt_reserved_attrs", "_pt_inner_ctx")

        def __init__(self, ctx):
            # validates no name collision with reserved attrs
            self._pt_inner_ctx = ctx

        def __getattr__(self, name):
            return getattr(self._pt_inner_ctx, name)

        def __setattr__(self, name, value):
            if name in type(self)._pt_reserved_attrs:
                self.__dict__[name] = value
                return
            return setattr(self._pt_inner_ctx, name, value)

Has two subclasses:

  CtxWithSavedTensors(WrappedCtx)
    Overrides saved_tensors property to return different tensors.
    Used when functorch needs to unwrap/rewrap tensors across levels.

  CtxCustomSave(WrappedCtx)
    Overrides save_for_backward and save_for_forward to unwrap batched
    tensors before saving. Used in vmap transforms.


Our _WrappedCtx
---------------
Overrides needs_input_grad only. Used in torch/_library/autograd.py to
transform the flat needs_input_grad tuple (strip metadata input, unflatten
to match user's input structure) without mutating the real ctx.

    class _WrappedCtx:
        def __init__(self, inner_ctx, needs_input_grad):
            self.__dict__["_inner_ctx"] = inner_ctx
            self.__dict__["_needs_input_grad"] = needs_input_grad

        @property
        def needs_input_grad(self):
            return self._needs_input_grad

        def __getattr__(self, name):
            return getattr(self._inner_ctx, name)

        def __setattr__(self, name, value):
            setattr(self._inner_ctx, name, value)


Key Differences
---------------
  Feature                    _WrappedCtx              WrappedCtx
  -------                    -----------              ----------
  Location                   torch/_library           torch/_functorch
  Reserved attrs mechanism   No                       Yes (_pt_reserved_attrs)
  __setattr__ behavior       Always forwards           Checks reserved, then forwards
  Name collision check       No                       Yes (raises on collision)
  What it overrides          needs_input_grad          Base for overriding anything
  Internal attr naming       _inner_ctx, _needs_...    _pt_inner_ctx, _pt_...
  Subclassable               Not designed for it      Yes (CtxWithSavedTensors, etc.)


Could we subclass functorch's WrappedCtx?
-----------------------------------------
Yes. It would look like:

    class CtxWithNeedsInputGrad(WrappedCtx):
        _pt_reserved_attrs = ("_pt_needs_input_grad", *WrappedCtx._pt_reserved_attrs)

        def __init__(self, ctx, needs_input_grad):
            super().__init__(ctx)
            self._pt_needs_input_grad = needs_input_grad

        @property
        def needs_input_grad(self):
            return self._pt_needs_input_grad

This works and follows the established pattern.


Should we combine?
------------------
Arguments for combining:
  - One wrapper pattern instead of two
  - Gets the reserved attrs mechanism for free (name collision validation)
  - Consistent _pt_ prefix naming convention
  - If we later need to override multiple attrs (e.g. needs_input_grad AND
    saved_tensors), the WrappedCtx base handles this naturally

Arguments against combining:
  - Couples torch/_library to torch/_functorch internals
  - _WrappedCtx is simpler (no reserved attrs overhead)
  - They serve different use cases: functorch wraps for transform levels,
    _library wraps for needs_input_grad transformation
  - The import path is awkward: a library feature importing from functorch

The coupling argument is the main one. torch/_library and torch/_functorch
are separate subsystems. If we want to share, the wrapper base class should
probably live in a neutral location (e.g. torch/autograd/function.py or
a shared utils module) rather than having one subsystem import from the other.

Recommendation: keep them separate for now. If a third wrapper appears,
extract the base class to a shared location at that point.


Should we worry about not having _pt_reserved_attrs?
----------------------------------------------------
The _pt_reserved_attrs mechanism solves two problems:

  1. Name collision detection: if the user sets ctx._pt_inner_ctx = foo
     in their forward, the functorch WrappedCtx raises an error in __init__
     rather than silently breaking.

  2. Correct __setattr__ routing: when the wrapper itself needs to store
     internal state (e.g. _pt_new_saved_tensors in CtxWithSavedTensors),
     __setattr__ checks the reserved list and stores in self.__dict__
     instead of forwarding to the inner ctx.

For our _WrappedCtx, neither problem applies:

  Problem 1 (name collision): Our internal attrs are _inner_ctx and
  _needs_input_grad. These are set in __init__ via self.__dict__ and
  never touched again. A user would have to set ctx._inner_ctx in their
  forward to collide, which is unlikely. And even if they did, it would
  not break silently -- __getattr__ would return the wrong object and
  the backward would likely error visibly. The functorch wrapper is more
  cautious because it's a general-purpose base class used across many
  transforms; ours is a single-purpose wrapper in one file.

  Problem 2 (__setattr__ routing): Our wrapper never sets internal attrs
  after __init__. The only property (needs_input_grad) is read-only.
  So __setattr__ can unconditionally forward to the inner ctx. Functorch
  needs the check because CtxCustomSave sets _pt_saved_tensors_bdims
  during backward (in save_for_backward), which must go to self.__dict__
  rather than the inner ctx.

So no, we should not be worried. The reserved attrs mechanism is needed
for the functorch wrapper's extensibility and mutability patterns, but
our wrapper is simple enough that it doesn't need it.
